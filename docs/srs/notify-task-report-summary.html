Notify Task Report — Tóm tắt logic luồng + Ghi chú quan trọng

Module: Assign (Giao việc) Phiên bản: 2.0 (Phase 12 + 13) Ngày: 2026-04-17 Trang nhập: /assign/tasks/daily-report Cron: assign:notify-task-report

1. Tổng quan 1 đoạn

Mỗi 30 phút cron chạy. Tới 4 mốc cố định 08:30 / 11:30 / 14:30 / 17:30, hệ thống quét các task IN_PROGRESS có rule báo cáo định kỳ active, đến lịch hôm nay nhưng chưa nhập tiến độ; gom theo assignee rồi gửi Web Push (FCM) tới browser. User click notification → mở thẳng trang /assign/tasks/daily-report → nhập tiến độ hàng loạt. Khi user nhập progress_pct > 0 ở mốc kế thì không bị nhắc nữa.

Các bước chi tiết

1Đăng ký FCM Token — User login → FE plugin fcm-push.js đăng ký SW firebase-messaging-sw.js → xin Notification.requestPermission() → messaging.getToken() → POST /assign/fcm-tokens {token, device_info}. BE updateOrCreate theo token.
2Cron trigger — Schedule everyThirtyMinutes()->withoutOverlapping() trong Kernel.php. Command tự gate: nếu currentMinutes ∉ {510, 690, 870, 1050} → log skip + return 0.
3Query task — Eager load Task IN_PROGRESS + progressReportRule.is_active=true + progressLogs đã filter report_date=today AND progress_pct>0 (bỏ N+1).
4Filter due today — Theo cycle_type: DAILY luôn true; WEEKLY today.dayOfWeek ∈ week_days; MONTHLY today.day == month_day.
5Skip đã nhập — Task có progressLogs.isNotEmpty() (sẵn pct > 0 hôm nay) → bỏ qua. Group còn lại theo assignee_id → đếm số task per user.
6Gửi FCM — Lấy tất cả token của user; build CloudMessage = Notification(title, body) + Data(click_action, task_count) + WebPushConfig(tag, fcm_options.link = FRONTEND_URL/assign/tasks/daily-report). Token NotFound → tự xoá khỏi fcm_tokens.
7User click — Background: SW custom notificationclick chạy trước Firebase + stopImmediatePropagation() → clients.openWindow(link) hoặc focus tab. Foreground: onMessage → toast trong app.
8Nhập tiến độ — Trang load qua GET /daily-report → render task chưa nhập trước (sort theo send_time). User edit chỉ dòng today → "Lưu tất cả" → FE validate sum past + today ≤ 100% → POST /daily-report/save.
9Lưu + sync — BE re-validate ownership (assignee_id == auth()->id()) + report_date == today + sum ≤ 100%. updateOrCreate task_result_progress_logs (key: task_id + report_date) → task.progress_pct = log mới nhất theo report_date.
10Mốc kế — Khi cron chạy mốc tiếp theo, task vừa nhập có progressLogs.isNotEmpty() → skip → user không bị nhắc nữa. Lặp đến hết 4 mốc/ngày.

4. ⚠️ GHI CHÚ QUAN TRỌNG

🚨 DEPLOY ORDER: Phải deploy code BE + FE TRƯỚC, sau đó mới chạy php artisan migrate. Nếu migrate trước → code cũ query whereNotNull('task_report_notify_time') trên column đã bị drop → fatal error.
🔒 HTTPS BẮT BUỘC: Web Push API + Service Worker chỉ hoạt động trên HTTPS (trừ localhost). Dev server HTTP → user không nhận được push. Đây là blocker đã ghi nhận từ Phase 11.
⚙️ ENV bắt buộc: FIREBASE_CREDENTIALS (path JSON service account) + FRONTEND_URL (URL tuyệt đối, vd https://erp.tpe.vn) — thiếu FRONTEND_URL → fcm_options.link sai → click notification không navigate đúng.
🕐 TIMEZONE: Carbon::now() dùng config/app.php timezone (đã set Asia/Ho_Chi_Minh). Nếu deploy môi trường khác mà timezone không phải VN → 4 mốc gửi sẽ lệch giờ.
📅 dayOfWeek convention: Carbon dùng 0=Chủ nhật ... 6=Thứ 7. Nếu DB lưu week_days theo convention khác (1=T2…7=CN) sẽ sai logic isReportDueToday cho cycle WEEKLY. Cần verify khi tạo rule mới.
🔁 Spam UX: Trước Phase 13 cron gửi 21 mốc/ngày. Hiện đã giảm xuống 4 mốc. Nếu tăng lại → user có nguy cơ tắt notification toàn site. Tag task-daily-report + renotify:true nghĩa là mỗi mốc đều kêu lại, không gộp.
🔐 Permission ownership: daily-report + save chỉ trả/chấp nhận task có assignee_id = auth()->id(). User A không thấy task của user B; ngay cả khi gửi task_id bậy, BE silent skip.
📊 Validate 100%: Sum progress_pct của tất cả log per task ≤ 100%. Validate cả FE (chặn submit) lẫn BE (skip task vi phạm + ghi errors, các task khác vẫn lưu).
🔄 Sync task.progress_pct: Sau lưu log, task gốc cập nhật = latestLog.progress_pct theo report_date lớn nhất (KHÔNG phải tổng cumulative). Nếu user nhập lùi ngày, vẫn lấy log có report_date lớn nhất.
♻️ Auto-cleanup token: FCM trả NotFound (token revoke/expired) → BE tự xoá row khỏi fcm_tokens → không retry vô hạn. Nhưng token vẫn cần được "refresh" định kỳ ở FE — nếu user không mở app lâu, token cũ có thể không còn valid.
✅ withoutOverlapping: Đã thêm trong Kernel để chống 2 instance cron chạy chồng (gửi double push). Lock mặc định 24h — nếu instance chết mà lock không release, lần kế bị block; trường hợp này phải xoá lock thủ công trong cache.
✅ Đã fix N+1: Eager load progressLogs đã filter today + pct>0 → loop trong PHP $task->progressLogs->isNotEmpty() không tốn query. Trước đó mỗi task gọi 1 query riêng.
✅ Skip đã nhập: Điều kiện hiện tại là progress_pct > 0. Nghĩa là user nhập 0% (chỉ giờ + ghi chú) sẽ vẫn bị tính là "chưa nhập" và bị nhắc lại ở mốc kế. Nếu cần đổi → review BR-03.
📅 Edge case MONTHLY: Rule cycle MONTHLY với month_day = 31 → tháng 30 ngày sẽ không bao giờ trigger. Đây là behavior cố định, không phải bug.
